# Piano B — Trovare CUDA, `nvcc`, cuBLAS e GPU A100

## B1. Controlla le GPU

```bash
nvidia-smi
```

Poi:

```bash
nvidia-smi --query-gpu=index,name,compute_cap,memory.total --format=csv,noheader
```

Per A100 devi vedere compute capability:

```text
8.0
```

Quindi per la build useremo:

```bash
export CUDA_ARCH=80
```

## B2. Trova `nvcc`

Prima prova:

```bash
which nvcc
nvcc --version
```

Se non lo trova:

```bash
find /usr/local /opt /storage -path "*/bin/nvcc" 2>/dev/null | sort
```

Esempi possibili:

```text
/usr/local/cuda/bin/nvcc
/opt/nvidia/hpc_sdk/Linux_x86_64/25.11/cuda/bin/nvcc
```

Se trovi `/usr/local/cuda/bin/nvcc`:

```bash
export CUDA_HOME=/usr/local/cuda
export PATH="$CUDA_HOME/bin:$PATH"
```

Se trovi, per esempio, HPC SDK:

```bash
export CUDA_HOME=/opt/nvidia/hpc_sdk/Linux_x86_64/25.11/cuda
export PATH="$CUDA_HOME/bin:$PATH"
```

Verifica:

```bash
which nvcc
nvcc --version
```

## B3. Trova `libcublas.so`

```bash
find /usr/local /usr/lib /opt /storage -name "libcublas.so*" 2>/dev/null | sort
```

Esempi possibili:

```text
/usr/local/cuda/lib64/libcublas.so
/usr/local/cuda/targets/x86_64-linux/lib/libcublas.so
/opt/nvidia/hpc_sdk/Linux_x86_64/25.11/math_libs/13.0/targets/x86_64-linux/lib/libcublas.so.13
```

Scegli la cartella che contiene `libcublas.so*`.

Esempio CUDA standard:

```bash
export CUBLAS_LIB=/usr/local/cuda/lib64
```

Esempio HPC SDK:

```bash
export CUBLAS_LIB=/opt/nvidia/hpc_sdk/Linux_x86_64/25.11/math_libs/13.0/targets/x86_64-linux/lib
```

Poi:

```bash
export LD_LIBRARY_PATH="$CUBLAS_LIB:$CUDA_HOME/lib64:${LD_LIBRARY_PATH:-}"
```

Verifica:

```bash
ls -lh "$CUBLAS_LIB"/libcublas.so*
```

## B4. Trova `cublas_v2.h`

```bash
find "$CUDA_HOME" /usr/local /opt /storage -name "cublas_v2.h" 2>/dev/null | sort
```

Esempio CUDA standard:

```bash
export CUBLAS_INCLUDE=/usr/local/cuda/include
```

Esempio alternativo:

```bash
export CUBLAS_INCLUDE=/usr/local/cuda/targets/x86_64-linux/include
```

Verifica:

```bash
ls -lh "$CUBLAS_INCLUDE/cublas_v2.h"
```

# Piano C — Build senza CMake

Da dentro il pacchetto:

```bash
cd /home/arnaud/jose/package_P_traincuda_integrated_v3_clean

export P_TRAINCUDA_ROOT="$PWD"
export PYTHONPATH="$PWD/python:${PYTHONPATH:-}"
export CUDA_ARCH=80
```

## C1. Build diretta con `nvcc`

```bash
rm -rf build
mkdir -p build

nvcc -O3 -std=c++17 -shared -Xcompiler -fPIC \
  -Icpp/include \
  -I"$CUBLAS_INCLUDE" \
  -gencode arch=compute_${CUDA_ARCH},code=sm_${CUDA_ARCH} \
  cpp/src/pomdp_backup_api.cu \
  -L"$CUBLAS_LIB" -lcublas \
  -o build/libpomdp_backup_cuda.so
```

## C2. Verifica che la `.so` esista

```bash
ls -lh build/libpomdp_backup_cuda.so
```

## C3. Verifica linking runtime

```bash
ldd build/libpomdp_backup_cuda.so | grep -E "cublas|cudart|cuda|not found"
```

Situazione buona:

```text
libcublas.so... => /path/to/libcublas.so...
libcudart.so... => /path/to/libcudart.so...
```

Situazione cattiva:

```text
libcublas.so... => not found
```

Se appare `not found`, rifai:

```bash
export LD_LIBRARY_PATH="$CUBLAS_LIB:$CUDA_HOME/lib64:${LD_LIBRARY_PATH:-}"
```

e ricontrolla:

```bash
ldd build/libpomdp_backup_cuda.so | grep -E "cublas|cudart|cuda|not found"
```

## C4. Test diretto da Python

```bash
python - <<'PY'
import ctypes
ctypes.CDLL("build/libpomdp_backup_cuda.so")
print("OK: CUDA backend library loaded")
PY
```

Se questo passa, la libreria è caricabile da Python.

## C5. Salva un file ambiente riutilizzabile

Crea questo file:

```bash
cat > .cuda_runtime_env.sh <<EOF
export P_TRAINCUDA_ROOT="$PWD"
export PYTHONPATH="$PWD/python:\${PYTHONPATH:-}"
export CUDA_HOME="$CUDA_HOME"
export CUBLAS_LIB="$CUBLAS_LIB"
export CUBLAS_INCLUDE="$CUBLAS_INCLUDE"
export LD_LIBRARY_PATH="$CUBLAS_LIB:$CUDA_HOME/lib64:\${LD_LIBRARY_PATH:-}"
export RF_LIB="$PWD/build/libpomdp_backup_cuda.so"
EOF
```

Ogni volta che riapri il terminale:

```bash
cd /home/arnaud/jose/package_P_traincuda_integrated_v3_clean
source .cuda_runtime_env.sh
```

# Diagnosi rapida degli errori più probabili

## Caso 1 — `nvcc: command not found`

Soluzione:

```bash
find /usr/local /opt /storage -path "*/bin/nvcc" 2>/dev/null | sort
```

Poi imposta `CUDA_HOME`.

## Caso 2 — `fatal error: cublas_v2.h: No such file or directory`

Soluzione:

```bash
find /usr/local /opt /storage -name "cublas_v2.h" 2>/dev/null | sort
export CUBLAS_INCLUDE=/path/che/contiene/cublas_v2.h
```

Poi ricompila.

## Caso 3 — `libcublas.so: cannot open shared object file`

La build è andata, ma runtime non trova cuBLAS.

Soluzione:

```bash
find /usr/local /usr/lib /opt /storage -name "libcublas.so*" 2>/dev/null | sort
export CUBLAS_LIB=/path/che/contiene/libcublas.so
export LD_LIBRARY_PATH="$CUBLAS_LIB:$CUDA_HOME/lib64:${LD_LIBRARY_PATH:-}"
```

Poi rilancia Jupyter dallo stesso terminale.

## Caso 4 — `CUDA_LIB exists: False`

Il notebook punta al pacchetto sbagliato.

Soluzione nel notebook:

```python
PATCH_ROOT = "/home/arnaud/jose/package_P_traincuda_integrated_v3_clean"
CUDA_LIB = os.path.join(PATCH_ROOT, "build", "libpomdp_backup_cuda.so")
```

## Caso 5 — `Environment.load(...)` fallisce

`ENV_PATH` è sbagliato o manca il dataset.

Soluzione:

```bash
find /storage /home -maxdepth 5 -type d -name "Env-320_485*" 2>/dev/null
```

Poi nel notebook:

```python
ENV_PATH = "/path/reale/del/dataset"
```

---

La cosa più importante: **non lanciare Jupyter prima di aver fatto `source .cuda_runtime_env.sh`**, perché altrimenti il notebook vede `traincuda`, ma quando arriva a caricare la `.so` può fallire su `libcublas.so`.
